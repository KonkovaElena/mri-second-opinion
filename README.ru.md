# MRI Second Opinion

[English](README.md) | Русский

**Система второго мнения по МРТ с обязательным участием врача.**

`MRI Second Opinion` — это отдельный TypeScript API и Python-воркер, которые ведут кейс МРТ от приёма и первичного контроля качества до формирования черновика заключения, обязательного врачебного разбора, финализации и передачи результата.

> **Важно. Только для исследований и инженерной оценки.** Репозиторий не является медицинским изделием и не может использоваться для клинических решений без регуляторного допуска. Любой результат должен проверяться квалифицированным врачом.

Репозиторий не пытается быть «ИИ, который читает МРТ». Это контур управления вокруг модели и вычислительного пайплайна, который принудительно удерживает человека в точке принятия решения.

## Что делает проект

- принимает кейс МРТ с метаданными исследования и псевдонимом пациента;
- проверяет доступные серии и подбирает подходящий маршрут обработки;
- запускает Python-воркер для вычислений и формирования артефактов;
- сохраняет результат модели как черновик, а не как финальный диагноз;
- требует обязательного врачебного просмотра перед финализацией;
- ведёт трассируемый журнал переходов, времени и артефактов;
- поддерживает экспорт финализированных результатов в DICOM SR и FHIR R4 DiagnosticReport.

## Что подтверждено текущей реализацией

- публичный API для приёма кейсов, разбора, финализации, просмотра отчёта и повторной доставки;
- внутренние маршруты для claim, heartbeat, callback, fail и retry;
- встроенное рабочее место разбора по пути `/workbench`;
- Python-воркер для стадий inference и delivery;
- локальная SQLite-база по умолчанию и задел для PostgreSQL;
- маршруты скачивания артефактов с поддержкой `local-file` и `s3-compatible` провайдеров;
- проверенная тестами и документацией граница между тем, что реализовано, что только планируется и что остаётся исследовательской гипотезой.

## Что проект не заявляет

- автономную диагностику;
- клиническую эксплуатацию без врача;
- завершённую интеграцию с PACS, OHIF или полноценным DICOMweb-контуром;
- готовность к регуляторному выводу на рынок;
- завершённую клиническую валидацию модели.

## Безопасность и эксплуатационная дисциплина

- внутренние маршруты закрыты Bearer-токеном `MRI_INTERNAL_API_TOKEN`;
- внешние операторские маршруты защищаются API-ключом `MRI_OPERATOR_API_TOKEN`;
- врачебные действия используют JWT HS256 с явной ролью и deny-by-default allowlist;
- HMAC-подпись применяется к маршрутам диспетчеризации воркера;
- входные данные валидируются Zod-схемами;
- используются ограничение частоты запросов, лимиты размера payload и timeout hardening на уровне сервера;
- ошибки возвращаются в структурированном JSON-формате без утечек стеков.

## Быстрый старт

Требования:

- Node.js 24+
- npm
- Python 3.11+ для воркера

```bash
npm ci
npm run build
npm test
npm start
```

Режим разработки:

```bash
npm run dev
```

После запуска полезно проверить:

- `http://localhost:4010/` — карта API;
- `http://localhost:4010/healthz` — базовый health check;
- `http://localhost:4010/workbench` — встроенный интерфейс для разбора кейсов.

### Docker

```bash
docker compose up --build app
docker compose up --build
```

### Запуск Python-воркера

```bash
export MRI_API_BASE_URL=http://127.0.0.1:4010
export MRI_INTERNAL_HMAC_SECRET=your-secret-key-at-least-32-bytes
export MRI_WORKER_STAGE=inference
python worker/main.py
```

## Проверка качества

Базовая локальная проверка:

```bash
npm test
```

В текущем наборе тестов покрываются жизненный цикл кейса, ограничения входных данных, отказоустойчивость интеграции с архивом, структура воркер-контракта, устойчивость PostgreSQL-пути, runtime hardening, работа с артефактами и проверка конфигурации.

Актуальный сводный снимок проверки зафиксирован в `docs/verification/release-validation-packet.md`.

## Технологический стек

- Node.js 24 и TypeScript 6.x для API и контуров управления;
- Express 5.x для HTTP-слоя;
- Zod 4.x для контрактной валидации;
- SQLite по умолчанию и PostgreSQL для промышленного хранилища;
- Python 3.11+ для воркера;
- Docker и Docker Compose для локального развёртывания и упаковки.
