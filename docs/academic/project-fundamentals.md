# Фундаментальное описание проекта MRI Second Opinion

Дата: 2026-03-31

Этот документ описывает **фундамент проекта**: технологический стек, архитектурные принципы, доменную модель и сравнительную позицию.

## Часть I. Технологический фундамент

### 1.1. Язык программирования

Проект написан на **TypeScript** — это надстройка над JavaScript, которая добавляет **строгую типизацию**. Что это значит простыми словами:

- JavaScript — самый популярный язык для веб-приложений. На нём работают все сайты.
- Его слабость — отсутствие проверок на этапе написания кода. Ошибки обнаруживаются только при запуске программы. В медицинском ПО это недопустимо.
- TypeScript решает эту проблему: компилятор **проверяет все типы данных до запуска**. Если программист напишет, что возраст пациента — текст вместо числа, код не скомпилируется.

**Почему TypeScript, а не Python или Java?**

| Язык | Плюсы | Минусы для данного проекта |
|---|---|---|
| **Python** | AI/ML библиотеки, научное сообщество | Медленнее для web-серверов, слабая типизация |
| **Java** | Строго типизирован, корпоративный стандарт | Тяжёлый, избыточен для standalone системы |
| **TypeScript** | Строго типизирован, лёгкий, один язык для backend + frontend | Нет нативного AI/ML (решается Python-воркером) |

Архитектурное решение проекта: TypeScript для **оркестрации** (управления workflow), Python для **вычислений** (нейросети). Каждый язык используется там, где он сильнее.

### 1.2. Среда выполнения

Программа работает на **Node.js** (версия 22+) — это среда, позволяющая запускать JavaScript/TypeScript-код на сервере (а не только в браузере).

Node.js выбран потому что:
- **Асинхронная модель** — сервер может обрабатывать сотни запросов одновременно без создания отдельного потока для каждого
- **Один язык** — и сервер (backend), и интерфейс врача (frontend) написаны на одном языке
- **Экосистема npm** — более 2 миллионов готовых пакетов (библиотек)

### 1.3. Веб-фреймворк

API построен на **Express.js** — минималистичном HTTP-фреймворке. Express — де-факто стандарт для Node.js-серверов. Он:
- принимает HTTP-запросы (от браузера, от нейросетей, от других систем)
- маршрутизирует их к нужным обработчикам
- возвращает ответы в формате JSON

### 1.4. Хранение данных

Проект поддерживает **два режима** хранения:

**SQLite** (по умолчанию) — база данных в одном файле:
- Не требует установки сервера баз данных
- Используется режим **WAL** (Write-Ahead Logging) — это означает, что чтение и запись могут происходить одновременно без блокировок
- Идеально для standalone-развёртывания (один врач, одна клиника)

**PostgreSQL** (опционально) — профессиональная серверная база данных:
- Для multi-user сценариев (несколько врачей одновременно)
- Поддерживает транзакции — группу операций, которые либо выполняются все вместе, либо не выполняются вообще (критически важно для медицинских данных)

## Часть II. Архитектура системы

### 2.1. Что такое архитектура ПО

Архитектура программы — это её **план**. Как архитектурный чертёж здания определяет, где стены, двери и окна, так архитектура ПО определяет:
- как части программы общаются между собой
- какие данные где хранятся
- какие правила нельзя нарушить

### 2.2. Архитектурный паттерн: Transparent Orchestrator

MRI Second Opinion использует паттерн **«Прозрачный оркестратор»**. Чтобы понять его, нужна аналогия:

**Обычный подход (чёрный ящик):**
```
Снимки → [?????] → Результат
```
Врач не знает, что произошло внутри. Не может проверить основу решения.

**Подход MRI Second Opinion (прозрачный оркестратор):**
```
Снимки → [Планирование] →? [Нейросеть #1] → [QC-проверка] → [Врач] → Результат
              ↕                   ↕                ↕             ↕
         Всё видно          Что нашла         Качество      Подтверждение
```

Каждый шаг:
1. **Видим** — врач знает, какой алгоритм был применён
2. **Проверяем** — есть отчёт о качестве исходных данных
3. **Контролируем** — без подписи врача результат не уходит

## Часть III. Доменная логика

### 3.1. Конечный автомат (стейт-машина)

Центральная концепция проекта — **конечный автомат** (finite state machine, FSM). Это математическая модель, в которой:
- система всегда находится в одном из **конечного набора состояний**
- переход между состояниями возможен только по **определённым правилам**
- правила **нельзя обойти**

В проекте определено **9 состояний** и строгие правила переходов:

```
INGESTING ──────→ SUBMITTED
                     │
            ┌────────┴────────┐
            ↓                 ↓
    QC_REJECTED        AWAITING_REVIEW
                            │
                            ↓
                        REVIEWED
                            │
                            ↓
                       FINALIZED
                            │
                            ↓
                     DELIVERY_PENDING
                     │              │
                     ↓              ↓
                 DELIVERED    DELIVERY_FAILED
                                    │
                                    ↓ (retry)
                             DELIVERY_PENDING
```

**Запрещённые переходы** (примеры):
- Нельзя перейти из SUBMITTED прямо в REVIEWED (нельзя пропустить ИИ-анализ)
- Нельзя перейти из AWAITING_REVIEW в DELIVERED (нельзя пропустить проверку врачом и финализацию)
- Нельзя перейти из DELIVERED обратно в REVIEWED (нельзя отозвать доставленный результат)

Эта модель гарантируется функцией `assertCaseStateInvariant()` — при любой попытке незаконного перехода система выбрасывает ошибку.

### 3.2. Clinician-in-the-loop (врач в контуре)

Это ключевой архитектурный принцип. Он означает:

> Ни один результат анализа ИИ не может покинуть систему без явного подтверждения квалифицированным врачом.

Это не настройка, а **инвариант** — свойство, которое система гарантирует на уровне кода. В `cases.ts` переход из `AWAITING_REVIEW` в `REVIEWED` требует:
- идентификатор рецензента (`reviewerId`)
- роль рецензента (`reviewerRole`)
- комментарий (`comments`)
- финальное впечатление (`finalImpression`)

### 3.3. Система планирования (Plan Envelope)

Когда снимки загружаются, система должна решить: **какие нейросети применить?**

Это происходит через **Plan Envelope** — структуру, которая описывает:
1. Какие последовательности МРТ доступны (T1w, FLAIR, DWI, SWI и др.)
2. Какие AI-пакеты подходят для этого набора
3. Какой пакет выбран (brain-structural, brain-lesion и др.)
4. Какова приоритизация workflow-веток (evidence branches)

Если обязательная последовательность (T1w) отсутствует — кейс автоматически переходит в `QC_REJECTED`.

### 3.4. Очереди задач (Job Queues)

Система содержит **две очереди**:

**Inference Queue** — задания для нейросетей:
- Создаётся при переходе кейса в `INFERENCE_RUNNING`
- Worker (Python-процесс) «берёт» задание через `claim-next`
- Механизм lease (аренды): если worker «упал», задание можно вернуть в очередь через `requeue-expired`

**Delivery Queue** — задания на доставку результата:
- Создаётся при финализации кейса
- Независимый delivery-worker отправляет результат
- При неудаче — статус `DELIVERY_FAILED`, можно повторить

Обе очереди **сохраняются при перезапуске** — это доказано тестами (тест создаёт кейс, останавливает сервер, запускает снова и проверяет, что данные на месте).

## Часть IV. Теоретические основы

### 4.1. Конечные автоматы (FSM) в медицинском ПО

Использование конечных автоматов для медицинских workflow — подход, подкреплённый академической литературой:

- **IHE (Integrating the Healthcare Enterprise)** использует конечные автоматы для описания транзакций в профилях интеграции
- **HL7 FHIR** определяет ресурс `Workflow` через стейт-машины (Task.status)
- **IEC 62304** (стандарт жизненного цикла медицинского ПО) рекомендует формальную верификацию состояний для критических систем

Проект идёт дальше: имеется **EFSM (Extended Finite State Machine) формальный анализ** — описание всех состояний, переходов и инвариантов в математической нотации (`docs/academic/formal-system-analysis.md`).

### 4.2. Модель честности заявлений (Claim Discipline)

Проект реализует **4-уровневую иерархию заявлений**:

| Уровень | Термин | Что значит |
|---|---|---|
| 1 | **Implemented** | Работает в текущем коде, проверено тестами |
| 2 | **Target** | Есть точки интеграции (seams) в коде, но функционал не реализован |
| 3 | **Research-Informed** | Обосновано научной литературой, но не начато |
| 4 | **Excluded** | Явно вынесено за пределы проекта, не обещается |

Это коррелирует с концепцией **Model Cards**, представленной FDA в январе 2025 в документе «AI-Enabled Device Software Functions: TPLC Guidance», где каждое ИИ-устройство должно иметь карточку с описанием возможностей, ограничений и данных обучения.

### 4.3. Regulatory-by-Design

Проект изначально спроектирован с учётом регуляторных требований:

| Принцип | Реализация | Регуляторная основа |
|---|---|---|
| Clinician-in-the-loop | Обязательная проверка врачом | FDA CDS Guidance (Jan 2026) |
| Transparency | Видимые findings, measurements, QC | FDA TPLC Model Cards (Jan 2025) |
| Traceability | operationLog + history array | IEC 62304 Maintenance |
| Supply-chain transparency | CycloneDX SBOM | FDA Section 524B (2025) |
| Threat awareness | STRIDE threat model | IEC 81001-5-1 |
| Bias awareness | Bias analysis framework | EU AI Act Art. 10 |

## Часть V. Практическое применение

### 5.1. Клинический сценарий использования

**Пример:** Пациент с жалобами на ухудшение памяти проходит МРТ головного мозга.

1. МРТ-снимки загружаются в систему (автоматически из PACS или вручную)
2. Система определяет: есть T1w и FLAIR → подходит пакет `brain-structural`
3. FastSurfer сегментирует структуры мозга. nnU-Net ищет патологии
4. Результат: гиппокамп уменьшен (z-score = −1.4), лёгкая генерализованная атрофия коры
5. Врач-нейрорадиолог открывает workbench, видит:
   - Findings: «Mild generalized cortical volume loss»
   - Measurements: hippocampal_z_score = −1.4
   - QC: motion artifact — warn (движение пациента, но интерпретация возможна)
6. Врач подтверждает: «No acute intracranial abnormality. Mild chronic volume loss»
7. Результат доставляется в исходную систему

### 5.2. Кому это нужно

| Аудитория | Зачем |
|---|---|
| **Нейрорадиологи в малых клиниках** | Доступ к ИИ-анализу без дорогих лицензий |
| **Академические исследователи** | Открытая платформа для валидации ML-моделей в клиническом контексте |
| **Телерадиологические сервисы** | Оркестрация второго мнения для удалённых пациентов |
| **Разработчики ИИ** | Готовая инфраструктура для подключения собственных моделей |

### 6.3. Ключевые отличия MRI Second Opinion

На март 2026 **ни один** из перечисленных проектов не сочетает:

1. **Принудительный clinician-in-the-loop** — в других системах проверка врачом опциональна или реализована через настройку. В MRI-SO она встроена в стейт-машину на уровне кода

3. **4-level claim discipline** — формальное разделение: что реализовано, что запланировано, что обосновано наукой, что исключено
4. **Низкий порог входа** — `npm install && npm run dev` вместо Kubernetes-кластера

## Часть VIII. Будущее проекта

### 8.1. Пять волн развития

```
Волна 1 (1-2 недели):    Hardening → rate limiting, graceful shutdown,
                           Prometheus-метрики
                              ↓
Волна 2 (3-4 недели):    Compute → реальный Python-воркер с FastSurfer
                           и nnU-Net для сегментации мозга
                              ↓
Волна 3 (2-3 недели):    Viewer → OHIF v3.12 + Orthanc для
                           профессионального просмотра снимков
                              ↓
Волна 4 (4-6 недель):    Interop → DICOM SR + FHIR export
                           для интеграции с больничными системами
                              ↓
Волна 5 (3-6 месяцев):   Evidence → клиническое исследование
                           (50-100 случаев, 2-3 нейрорадиолога)
```

### 8.2. Регуляторный путь

Наиболее вероятный путь: **FDA CDS Non-Device**.

FDA в январе 2026 расширила категорию ПО, которое **не является медицинским устройством**: если программа даёт **одну клинически обоснованную рекомендацию**, а врач может **независимо проверить её основу** — регистрация как медизделие не требуется.

MRI Second Opinion попадает в эту категорию, потому что:
- Выдаёт второе мнение, а не диагноз
- Показывает основу заключения (findings, measurements, QC)
- Врач обязан проверить и подтвердить результат

### 8.3. Научная публикация

Целевая стратегия:
1. Препринт на **arXiv** (cs.CV + eess.IV)
2. Постер на **RSNA 2027** или **ECR 2027**
3. Оригинальная статья в **Radiology: Artificial Intelligence**
