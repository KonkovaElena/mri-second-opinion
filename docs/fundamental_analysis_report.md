# Анализ фундаментальной части и сравнение с конкурентами

> Для эпидемиологического обоснования, когнитивных предвзятостей, моделей реализации и стратегического синтеза см. `docs/academic/deep-academic-analysis.md`.

## 🧱 Фундаментальная часть проекта (Core Architecture)

Фундаментом проекта **MRI Second Opinion** является шаблон **"Прозрачного Оркестратора" (Transparent Orchestrator)**, написанный на TypeScript. Проект намеренно отказывается от создания очередного "черного ящика ИИ" в пользу легковесного, управляемого событиями (event-driven) контроллера.

**Основные принципы фундамента:**
1. **Разделение управления и вычислений:** Web API (TypeScript) отвечает только за логику состояний, маршрутизацию и формирование отчета. Тяжелая обработка снимков (Python) и их хранение (Orthanc/MinIO) полностью вынесены в отдельные специализированные узлы.
2. **Осознанная маршрутизация (Sequence-aware routing):** Снимки не просто "кидаются в нейросеть". Оркестратор сначала проверяет наличие нужных последовательностей (например, T1w, FLAIR) в метаданных DICOM и только потом подбирает подходящий "пакет" рабочих алгоритмов (workflow package) для анализа.
3. **Двухуровневая ИИ-стратегия (Foundation + Specialist):** 
   - *Уровень 1:* Базовые "Foundation" модели (такие как MedSAM2) используются только для поверхностного скрининга аномалий.
   - *Уровень 2:* Узкоспециализированные пайплайны (nnU-Net, FastSurfer, SCT) используются для точных количественных измерений конкретных патологий (объемы желудочков, очаги поражения).
4. **Событийная модель очередей:** Вместо пакетной обработки всей базы по расписанию, триггером выступает поступление нового снимка в PACS, которое моментально ставит задачу в Redis-очередь (в данном проекте - BullMQ).

## 🥊 Сравнение с аналогами и конкурентами в экосистеме

Архитектура проекта была сформирована с оглядкой на существующие mature-решения (согласно документам на Март 2026). Вот как проект позиционирует себя относительно конкурентов:

### 1. Архитектурные платформы и Оркестраторы

| Платформа-Аналог | Суть решения конкурента | Позиция MRI Second Opinion |
| :--- | :--- | :--- |
| **Kaapana** *(0.6.x)* | Тяжелая платформа для управления жизненным циклом ИИ-образов, Kubernetes-first, глубокое управление доступом. | **Легковеснее:** Проект отказывается от перегруженной платформенной среды Kaapana. Он нацелен на узкий самодостаточный MVP из 7 узлов (без обязательного Kubernetes на старте), оставляя Kaapana лишь как референс для будущих enterprise-внедрений. |
| **MONAI Deploy (App SDK)** | Стандарт де-факто для создания DAG-пакетов (MAPs) для медицинских моделей и их развертывания. | **Независимее:** Проект поддерживает совместимость с подходами MONAI (внутри Python worker), но не делает SDK MONAI Deploy обязательным диктатором всего пайплайна. Управление остается на стороне собственного TypeScript API. |
| **XNAT** *(1.9.x)* | Мощный репозиторий для исследовательских медицинских данных с расширенными функциями по созданию когорт и анонимизации. | **Клинический фокус:** XNAT — это серьезный исследовательский комбайн. MRI Second Opinion — это сфокусированный клинический маршрутизатор второго мнения. Интеграция с XNAT рассматривается как будущая фича для научных нужд, но не как ядро. |
| **Apache Airflow** *(3.x)* | Индустриальный стандарт для запуска пакетных (batch) задач по расписанию (DAGs). | **Событийно-ориентированнее:** Получение МРТ снимка пациента — это событие (event streaming), для которого Airflow подходит плохо. Референсом для проекта здесь выступает паттерн *Prefect* или прямая нативная реализация поверх Redis. |

### 2. Подходы к Моделям и движкам ИИ

| Модный подход на рынке | В чем подвох | Решение в MRI Second Opinion |
| :--- | :--- | :--- |
| **"Один Medical LLM/VLM для всего"** *(LLaVA-Med, Med42)* | Большие языковые и мультимодальные модели могут галлюцинировать. В лицензиях часто прямо запрещено их клиническое использование. | **LLM только как ассистент.** Структурированные измерения (json, маски, точные объемы) — единственный источник правды. Генерация текста (VLM) — применяется строго как помощь врачу ("review-assist"), а не замена диагноза. |
| **Только Foundation Models** *(MedSAM2 и аналоги)* | Лицензии весов Foundation моделей (HuggingFace) часто сугубо академические (research-only). Дополнительно, их точность в задачах вроде сегментации спинного мозга может уступать проверенным узким сетям. | **Симбиоз.** Проверенные узкие CNN (nnU-Net v2.6.4, FastSurfer) остаются рабочей лошадкой для точных замеров. Foundation модели могут подключаться только в качестве страховочной сети для поиска аномалий. |

## 🧭 Итоговый вывод
Проект **MRI Second Opinion** не пытается стать очередной неподъемной PACS-системой, "новым MONAI" или исследовательским комбайном вроде **XNAT/Kaapana**. 

Его "киллер-фича" — **прозрачность, модульность и честность**. 
Это узкий, легковесный маршрутизатор (API на TypeScript + Очереди Redis), который выступает интеллектуальным прокси: он принимает снимок, отсеивает брак (QC), подбирает под снимок лучшую узкоспециализированную нейросеть (из зоопарка Python моделей с открытым исходным кодом), агрегирует метрики и формирует удобочитаемую карточку "Второго мнения" для реального врача на базе просмотрщика OHIF, ни на одном из этапов не пытаясь занять место самого врача.
